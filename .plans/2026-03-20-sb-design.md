# `sb` — Sandbox Manager

## Motivation

AI coding agents like Claude Code are powerful but risky when given direct access to the host filesystem and network. An agent executing a plan can modify files, install packages, or make network calls — all on your actual machine. Mistakes, hallucinations, or unexpected behavior can have real consequences.

Lima VMs provide lightweight Linux isolation on macOS: the agent operates in a sandboxed environment where it can freely read/write files, install tools, and run services without touching the host. By mounting specific host directories into the VM and maintaining path parity between host and VM, the agent's workspace looks identical from both sides — paths resolve the same way, and work done in the VM is immediately visible on the host (and vice versa) for mounted directories.

But managing this manually — creating VMs, configuring mounts, copying dotfiles, installing toolchains — is repetitive ceremony that scales poorly when you're running multiple agent tasks.

`sb` reduces that to `sb create` and `sb shell`. One command to spin up a provisioned Linux VM with your mounts and config files. One command to enter it. It's unopinionated about what's installed inside — you provide setup scripts for your toolchain, and `sb` handles the VM lifecycle and provisioning.

## Key Design Principles

- **Path parity**: The VM user has the same username, UID, GID, and home directory as the host. Mounts preserve host paths (same source and mount point). Plain `copy_paths` entries copy to the same location. All absolute paths resolve identically on both sides.
- **Unopinionated VM contents**: `sb` manages the VM lifecycle and provisioning mechanics. What gets installed (Docker, Go, Node, etc.) is controlled by user-provided setup scripts, not hardcoded in the tool.
- **Single VM**: One sandbox, named `sb`. Keeps the CLI simple and matches the primary use case of one isolated environment per machine.

## Commands

| Command | Purpose |
|---------|---------|
| `sb create` | Create VM, start it, copy paths, run scripts. Re-provisions if already running. Starts + provisions if stopped. |
| `sb start` | Start a stopped VM. Error if not created. |
| `sb stop` | Stop a running VM. No-op if already stopped or not created. |
| `sb destroy [--force]` | Stop (if running) and remove the VM. Requires `--force` or interactive confirmation. |
| `sb status` | Print VM state: `running`, `stopped`, or `not created`. |
| `sb provision` | Re-copy paths and re-run scripts on a running VM. Error if not running. |
| `sb shell [-- command]` | Open interactive shell, or run a command in the VM. |
| `sb config path` | Print config file path. |
| `sb config edit` | Open config in `$EDITOR`. Creates default if missing. |
| `sb config refresh` | Create or update config with defaults for new fields. |
| `sb completion [bash\|zsh\|fish]` | Generate shell completions. |
| `sb --verbose` | Global flag for debug output. |

## Configuration

Single file at `~/.config/sb/config.json`:

```json
{
  "image": "ubuntu-24.04",
  "cpus": 4,
  "memory": "4GiB",
  "disk": "100GiB",
  "mounts": [],
  "copy_paths": [],
  "scripts": []
}
```

- **`image`** — Lima base image identifier. Default `"ubuntu-24.04"`.
- **`cpus`** — VM CPU count. Default `4`.
- **`memory`** — VM memory. Default `"4GiB"`.
- **`disk`** — VM disk size. Default `"100GiB"`.
- **`mounts`** — Host paths to mount read-write into the VM at the same path. Example: `["/Users/avery/work"]`.
- **`copy_paths`** — Files to copy into the VM. Plain path copies to the same location in the VM. `src:dst` syntax maps to a different destination. Example: `["~/.zshrc", "~/.claude/sandbox/settings.json:~/.claude/settings.json"]`.
- **`scripts`** — Host paths to scripts that run inside the VM during provisioning. Executed in order. Example: `["~/.config/sb/scripts/docker.sh", "~/.config/sb/scripts/devtools.sh"]`.

Auto-created with defaults on first run of any command that needs it.

## Paths & Naming

| Resource | Path |
|----------|------|
| Config file | `~/.config/sb/config.json` |
| VM name | `sb` (used with limactl) |

All paths follow XDG conventions (`XDG_CONFIG_HOME`). No data directory needed — Lima manages VM state in its own `~/.lima/` directory.

## Command Flows

### `sb create`

1. Load config (create default if missing).
2. Check VM status via limactl.
3. If running: run provision (copy paths + scripts) and return.
4. If stopped: start, then provision.
5. If not created: render lima.yaml template with config values (image, cpus, memory, disk, mounts, user/UID/GID/home from host), create VM via limactl, then provision.

### `sb provision`

1. Error if VM is not running.
2. For each entry in `copy_paths`: create parent directory in VM, then copy file via `limactl cp`. Plain paths copy to the same location; `src:dst` entries map accordingly.
3. For each entry in `scripts`: copy script to a temp path in the VM, execute it, remove temp file.

### `sb shell`

- No args: `limactl shell --workdir / sb`
- With command: `limactl shell --workdir / sb -- <command>`

### `sb destroy`

1. Check VM status. No-op if not created.
2. If running, stop first.
3. Delete via `limactl delete sb`.
4. Requires `--force` flag or interactive confirmation.

### `sb start`

1. Check VM status.
2. Error if not created (requires `sb create` first).
3. No-op if already running.
4. Otherwise start via `limactl start sb`.

### `sb stop`

1. Check VM status.
2. No-op if not created or already stopped.
3. Otherwise stop via `limactl stop sb`.

### `sb status`

1. Check VM status via limactl.
2. Print `running`, `stopped`, or `not created` to stdout.

### Dependency Injection

```go
type Service struct {
    lima   limaClient
    config config.Config
    logger *slog.Logger
}
```

The `limaClient` interface wraps all limactl operations, making the service fully testable without a real VM.

### Lima Template

Minimal embedded YAML template rendered at create time:

- Base image from config (`image`)
- CPU / memory / disk from config
- User mapping from host (username, UID, GID, home directory — preserves path parity)
- Mounts from config (same source and mount point, read-write)
- containerd disabled
- No provisioning scripts in the template — all software installation happens through user-provided `scripts`

### Error Handling

- All Cobra commands use `RunE` (not `Run`) to return errors.
- Root command sets `SilenceUsage: true` and `SilenceErrors: true` — runtime errors don't dump usage text.
- `main.go` handles the top-level error: log it with `slog.Error` and `os.Exit(1)`.
- Wrap errors with context using `%w`: `fmt.Errorf("failed to create VM: %w", err)`.

### Logging

Uses `log/slog` from the standard library. Verbose flag controls log level (default `Warn`, verbose `Debug`).

### Testing

- Unit tests use mock implementations of interfaces (`testify/mock`).
- Test naming: `TestType_Method_Scenario`.
- Always run with race detection: `go test -race ./...`.

## What We're NOT Building

- No multiple VM support — single VM named `sb`
- No built-in provisioning scripts — users provide their own via `scripts`
- No template subcommand — debug the rendered YAML by reading Lima's files directly
- No network policy configuration — Lima defaults apply
- No snapshot/restore — out of scope
